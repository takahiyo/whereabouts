# ログイン時の誤入室（ID: elenia）デバッグ記録

## 現象
Firebaseで登録したメールアドレスを使用してログインした際、新規開設拠点作成ではなく、D1上の既存拠点（ID: elenia）の管理者としてログインしてしまう。

## 調査状況と原因の特定
- `auth.js` の `handleFirebaseUser` と、 `CloudflareWorkers_worker.js` の `signup` アクションを確認しました。
- **原因:** 先日の修正で追加した「Firebase側でユーザーが再作成された場合に備え、`email`が一致すればD1の既存ユーザー（UIDを更新して）と再紐付けする機能」が働いています。
- D1の `users` テーブルにおいて、このメールアドレス（taka.hiyo@gmail.com）にすでに `office_id = 'elenia'` が登録されているため、システムは「既存のオーナーがログインした」と判定し、自動的に `elenia` にログインさせています。（DBの末尾にあるからではなく、メールアドレスで紐付いているためです）

## 解決へのアプローチ案
このメールアドレスで「新規拠点作成」画面に進むための対応として、以下のいずれかが必要です。
1. **D1データベース上のデータ修正（手動）：**
   D1の `users` テーブルから該当のメールアドレスのレコードを削除、または `office_id` を一旦 NULL にする。
2. **システム仕様の変更（自動対応）：**
   もしオーナーが複数の拠点を管理できるようになっている、あるいは「既存拠点にログインするか、新しい拠点を作成するか」選べるようにしたいのであれば、`auth.js` 側で `resp.user.office_id` があっても強制的に `createOffice` も選べるUIに変える。ただし現在は「1ユーザー＝1拠点（管理）」の前提で組まれているようです。

まずは、ユーザーの目的が「テストのために一旦 `elenia` との紐付けを解除して新規作成画面を出したい」のか、「1つのアカウントで複数の拠点を作成・管理したい」のかを確認する必要があります。

## 確認事項
1. `finalizeLogin` に渡される `office` がどのように決定されているか（`handleFirebaseUser`付近）
2. Worker側の `/auth/verify` または `login` で、既存の拠点が存在しない場合にもかかわらず `office=elenia` を返却していないか
