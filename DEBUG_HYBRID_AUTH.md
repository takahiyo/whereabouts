# ハイブリッド認証実装 デバッグログ (2026-04-17 更新)

## 現在発生している問題
- **現象**: 
    1. Firebaseでログイン後、ブラウザに過去の拠点キャッシュが残っていると、その拠点の管理者として入れてしまう（認可の不備）。
    2. 新規拠点が未開設の状態でも、過去の拠点名（`verllion`等）が表示されてしまう。
- **調査ログ**:
    - `auth.js` V5-V6: `finalizeLogin` 呼び出し時に `token=false` となるケースがあった。
    - Worker側の `renew` がトークンを返していなかったため、フロントエンドでセッションが途切れていた。
    - 根本原因として、Worker側で「リクエストされた拠点ID」と「ログインユーザーの所属拠点」の不一致をチェックする認可ロジックが欠落していた。

## 原因の仮説 (2026-04-17 追加)
1. **認可ガードの欠如**: Workerはトークンさえ正しければ（どの拠点のものであっても）、パラメータで指定された拠点のデータにアクセスさせてしまっていた。
2. **キャッシュの固着**: `localStorage` に保存された `presence-office-id` や `officeName` が、ログアウトやユーザー切り替え時に不完全にしかクリアされていなかった。

## 実施した対策とデバッグ強化 (V7)
1. **[Worker] 認可ガードの導入**:
    - すべてのアクションに対し、`superAdmin` でない限り「リクエストされた拠点」と「トークンに紐付いた拠点」が一致するかを検証する共通処理を追加。
    - 不正なアクセスには `403 Forbidden (office_access_denied)` を返すように強化。
2. **[Frontend] キャッシュクリーニングの徹底**:
    - `logoutAction` および `handleWorkerError` において、`officeName` を含むすべての関連キャッシュを完全に削除。
    - サーバーから `office_id` が `null` で返ってきた場合、即座にローカル情報を破棄するように修正。
3. **[Frontend] 認可エラーハンドリング**:
    - `403` エラー（拠点アクセス拒否）を受け取った際、アラートを表示してキャッシュを初期化した後、リロードする処理を追加。
4. **バージョンの引き上げ**:
    - `index.html` および各ファイルのバージョンを **v7** に更新。
