## 2026-03-31: 拡張カラム（伸びる列）による他カラム圧縮問題の修正

### 成功の境界線
- 備考等の「最大幅の制約がないカラム」が存在する場合において、指定幅が設定されている他のカラム（氏名、業務時間等）が本来の指定幅を維持できるようになり、画面全体のレイアウト崩れが解消された。

### 失敗の事象
- 以前の修正で、最大幅を持たない伸縮カラムに対して強制的に `width: 100%` を付与していた。
- `.panel table` 自体が `width: 100%` を持っている状態で、単一の `td/th` に `100%` の幅を要求すると、`table-layout: auto` のレンダリングアルゴリズムがその要求を最優先し、固定幅指定の他カラムを強制的に最小幅（`min-width` や内容の最小幅）まで圧縮（squish）してしまうという甚大なレイアウト崩れが発生した。

### 失敗の根本原因
- 標準のHTMLテーブルにおいて、残りの余白を埋めさせる目的で `width: 100%` を使用したこと。フレックスボックス（Flexbox）とは異なり、テーブルのセルにおける `100%` は「可能な限り親要素と同じ幅を要求する」という強欲な指定となるため。

### 次のアプローチ（今回の解決策）
- `width: 100%` の指定を廃止し、未指定の列と同様に `width: 'auto'` を指定するように緩和する。
- テーブル全体が `width: 100%` を持っているため、固定幅を持たない `auto` カラムは、他の固定幅カラムの描画後に「自然に残ったスペースをすべて吸収する」という正しいテーブルレイアウトの挙動となる。

## 2026-03-31: 500エラー解消と幅制約 (max-width) 反映の不具合修正

### 成功の境界線
- [x] `js/board.js` の修正（ストレッチ列 `100%` と制限列 `px` の明示的指定）
- [x] `CloudflareWorkers_worker.js` の修正（`getEventColorMap` 等の `try-catch` 追加）
- [x] Browser Agent による動作検証（業務時間が 50px 程度まで縮小されるか）
- [x] 開発ログ (`dev_log.md`) の更新
- `styles.css` から `table-layout: fixed` を除去し、`table-layout: auto` に変更することで、カラムへの `max-width` 指定が有効に機能するようになった。
- `CloudflareWorkers_worker.js` の `setColumnConfig` を `try-catch` で保護し、エラー時に 500 エラーではなく詳細なバックエンド情報を返却できるようになった。
- `js/board.js` の `applyWidthStyle` を `auto` レイアウト環境下で最適に動作するように調整（固定幅以外は `width: auto` + `min/max-width`）。

### 失敗の事象（解決済み）
- カラムの最大幅を 50 に設定しても画面が変化しない → `fixed` レイアウトが `max-width` を無視していたことが原因。
- 拠点設定の保存時に 500 エラーが発生 → Worker側の例外処理不足により、テーブル不在等のエラーでクラッシュしていた。

### 失敗の根本原因
- 従来の `table-layout: fixed` 設定は均等割り付けには向いていたが、今回導入した「柔軟な最小・最大幅指定 (Requirement 3)」とは技術的に相反するものであった。
- Worker の `setColumnConfig` ハンドラが DB 操作に対してフェイルセーフな設計になっていなかった。

### 次のアプローチ
- カラム幅を厳格に制御したい場合は `min == max` を指定し、内容に応じて変動させたい場合は `auto` レイアウトの特性（`min/max-width`）を最大限活用するようにユーザーに案内する。

## 2026-04-08: QRコード生成エラー（bad rs block）の修正

### 成功の境界線
- ログイン後の「📱 QRcode」表示において、拠点IDを含むURLが正常にQRコード画像（SVG）として生成・表示されるようになった。
- エラー訂正レベルの指定が文字列（'M'など）であっても、内部で正しく数値定数にマッピングされるようになり、ライブラリの参照エラーが解消された。

### 失敗の事象
- QRコード生成時に `QR Generation failed: Error: bad rs block @ typeNumber:1/errorCorrectionLevel:M` というエラーがコンソールに出力され、画像が表示されなかった。

### 失敗の根本原因
- `js/services/qr-generator.js` 内の `getRsBlockTable` 関数が、エラー訂正レベルを数値定数（L=1, M=0...）で厳密比較していたのに対し、`auth.js` からは文字列 `'M'` が渡されていたため、テーブル参照が `undefined` となりクラッシュしていた。
- また、使用していたライブラリの `RS_BLOCK_TABLE` が Version 10 までしか定義されていなかったが、自動判別ロジックが Version 40 まで試行しようとしており、長大なデータ入力時に境界外参照を起こすリスクがあった。

### 次のアプローチ（今回の解決策）
- `qr-generator.js` の `getRsBlockTable` に文字列から数値へのマッピング処理を追加。
- バージョン自動判別（`setTypeNumber(0)`）時のループ範囲を、実際に定義されているテーブル長（Version 10）までに制限し、超過時には適切なエラーを投げるようにガードを追加。
